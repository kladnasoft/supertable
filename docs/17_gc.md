# 17. Storage Garbage Collection

## Overview

Sunset parquet files (replaced during compaction) and old snapshot
JSONs are never deleted by `DataWriter` directly. Instead they're
pushed onto a **per-table Redis STREAM** after a successful leaf-CAS
commit. A separate orchestrator (`GCCleaner.tick()`) drains entries
older than a configurable delay window and calls `storage.delete()`.

The delay window is the safety guarantee: by the time the cleaner
touches a file, any in-flight reader that resolved the leaf payload
before the writer committed has long since finished. We never delete
a file that's still reachable from any reader's plan.

**The SDK ships no daemon for this** — `tick()` is a pure
orchestration primitive that your service calls on its own schedule.
A convenience `python -m supertable.gc.daemon` wrapper exists for
single-node installs without a scheduler.

## Off By Default

| Variable | Default | When set |
|---|---|---|
| `SUPERTABLE_SUNSET_GC_ENABLED` | `false` | Sunset parquet files queued for deletion |
| `SUPERTABLE_SNAPSHOT_RETENTION` | `0` (unlimited) | Snapshot JSONs older than the `N`-th most recent are queued for deletion |
| `SUPERTABLE_GC_DELAY_SEC` | `1800` (30 min) | Minimum age before the cleaner deletes |
| `SUPERTABLE_GC_SLEEP_SEC` | `60` | Cleaner-daemon loop sleep (convenience wrapper only) |
| `SUPERTABLE_GC_BATCH_SIZE` | `500` | Max entries per stream per tick |

With both feature flags off (the default) the writer makes **zero**
extra Redis calls and behaviour is identical to a pre-GC SDK build.

## Producer Side — `DataWriter`

Immediately after the leaf-CAS + root-bump in `data_writer.py`, the
writer adds:

```python
from supertable.gc.queue import collect_old_snapshot_paths, enqueue_deletions

try:
    if settings.SUPERTABLE_SUNSET_GC_ENABLED and sunset_files:
        enqueue_deletions(
            self.catalog, org, sup, simple_name,
            "parquet", list(sunset_files), write_id=qid,
        )
    if settings.SUPERTABLE_SNAPSHOT_RETENTION > 0:
        old_paths = collect_old_snapshot_paths(
            new_snapshot_dict, self.super_table.storage,
            settings.SUPERTABLE_SNAPSHOT_RETENTION,
        )
        if old_paths:
            enqueue_deletions(
                self.catalog, org, sup, simple_name,
                "snapshot", old_paths, write_id=qid,
            )
except Exception as e:
    # GC failure must never fail a write.
    logger.warning(f"GC enqueue failed (write still succeeded): {e}")
```

**Strict ordering matters.** The XADD must happen *after* the
leaf-CAS, never before. If XADD landed first and the CAS then failed
in a retry, the stream would point at files still referenced by the
live snapshot and the cleaner would silently delete data. The
reverse order — CAS first, XADD second — has only one failure mode
(writer crash between the two) which leaks files without losing any.
That's identical to the pre-GC behaviour.

### Stream layout

```
supertable:{org}:lakes:{sup}:gc:pending:doc:{simple}       STREAM
```

Per-(super, simple). Entries are simple `{kind, path, write_id?}`
records:

```
XADD <stream> *
    kind     "parquet" | "snapshot"
    path     <storage path>
    write_id <qid>            ← optional, for tracing
```

The natural Redis Stream ID (`<ms>-<seq>`) carries the enqueue
timestamp, so the cleaner enforces the delay window via `XRANGE -
<cutoff_ms>-0` with no per-entry timestamp field.

### Lifecycle integration

| Operation | Effect on GC stream |
|---|---|
| `DataWriter.write()` succeeds | XADD per sunset / pruned-snapshot path |
| `SimpleTable.delete()` | `RedisCatalog.delete_simple_table()` `DEL`-s the stream (plus leaf + lock) |
| `SuperTable.delete()` | `RedisCatalog.delete_super_table()` SCANs `super_table_pattern` and `DEL`-s everything — the GC stream is naturally swept too |

## Snapshot Retention Math

`SUPERTABLE_SNAPSHOT_RETENTION=N` means "keep the **N** most recent
snapshot JSONs per simple table". The writer walks the new snapshot's
`previous_snapshot` chain via `collect_old_snapshot_paths()`:

- Position 0 = the snapshot just written (kept).
- Positions 1..N−1 = preceding snapshots in the chain (kept).
- Position N and beyond = queued for deletion.

The walk reads predecessor JSONs from storage one step at a time;
missing JSONs (already pruned earlier) and detected cycles stop the
walk gracefully — the function never raises.

The walker is safety-capped at 10 000 hops as defence against
corrupted chains.

## Consumer Side — `GCCleaner.tick()`

The library surface (`supertable/gc/cleaner.py`) is **pure
orchestration**: no loops, no threads, no signal handlers. One
`tick()` is one full pass over the org's per-table GC streams.

```python
from supertable.gc.cleaner import GCCleaner
from supertable.redis_catalog import RedisCatalog
from supertable.storage.storage_factory import get_storage

cleaner = GCCleaner(
    org="acme",
    catalog=RedisCatalog(),
    storage=get_storage(),
    # Optional — defaults come from SUPERTABLE_GC_* env vars
    delay_sec=1800,
    batch_size=500,
)

# Caller-owned scheduling — your service decides when to run
while not service.shutdown_requested:
    stats = cleaner.tick()
    service.publish_metrics(stats)
    service.sleep(60)
```

### What one `tick()` does

1. SCAN `supertable:{org}:lakes:*:gc:pending:doc:*` for every stream
   in the org.
2. For each stream, `XRANGE - <cutoff>-0 COUNT batch_size` (`cutoff_ms = now − delay_sec * 1000`).
3. For each returned entry, `storage.delete(path)`:
   - On success → record in stats and add the entry id to the XDEL batch.
   - On `FileNotFoundError` → entry id still queued for XDEL (idempotent — already gone).
   - On any other exception → leave the entry on the stream so the next tick retries.
4. `XDEL` the processed ids (in one batch per stream).

### Return value

```python
{
    "streams_processed": int,    # streams the tick inspected without error
    "deleted":           int,    # total files removed this tick
    "deleted_parquet":   int,    # of which parquet
    "deleted_snapshot":  int,    # of which snapshot JSON
    "errors":            int,    # per-stream XRANGE failures (one per stream)
}
```

Surface these to your service's metrics pipeline — a sustained
`errors` count indicates a Redis hiccup or a misbehaving storage
backend.

### Concurrency

`tick()` is safe to call concurrently from multiple processes. The
worst case is two cleaners XDEL-ing the same entry (Redis tolerates
this) and double-attempting a `storage.delete` (also idempotent via
`FileNotFoundError`). There's no Redis-level locking required.

### Stopping mid-tick

`cleaner.stop()` sets an internal flag that's consulted between
streams. A long-running tick (large batch, many streams) bails at
the next stream boundary. To reuse the cleaner after a stop,
instantiate a new one — the flag is not auto-reset (cheap; no
persistent state).

## Convenience Daemon — `supertable.gc.daemon`

For deployments without their own scheduler:

```bash
# Single org
python -m supertable.gc.daemon --org acme

# Multi-tenant (discovers orgs via SCAN, spawns one thread each)
python -m supertable.gc.daemon --all-orgs

# Overrides
python -m supertable.gc.daemon --org acme \
    --delay-sec 3600 --sleep-sec 30 --batch-size 200
```

Internally:

```python
def run_forever(cleaner: GCCleaner) -> None:
    while not cleaner._stop.is_set():
        try:
            stats = cleaner.tick()
            ...
        except Exception as e:
            logger.error(...)   # daemon never dies
        time.sleep(cleaner.sleep_sec)
```

`run_all_orgs(...)` spawns one `run_forever` thread per discovered
org and re-scans periodically (`--discover-every-sec`, default 300)
to pick up newly-active orgs without a restart.

SIGINT and SIGTERM trigger a graceful shutdown: cleaners stop at
their next stream boundary, threads are joined with a 10-second
timeout.

**The convenience daemon is genuinely optional.** Most production
deployments should call `tick()` from their own service loop — they
get free integration with their existing scheduler, metrics, logging,
and shutdown handlers.

## Reader Race — Why the Delay Window?

Without GC, a query plan can stay valid for the duration of a long
SQL execution: the reader resolves the leaf payload, gets a list of
parquet paths, and DuckDB's `parquet_scan([...])` reads them. Now a
compaction commits a new snapshot that excludes some of those paths
— and immediately tries to delete them. The in-flight reader's
parquet open fails with `FileNotFoundError` (DuckDB does not have
the `_read_parquet_safe` retry that the writer uses internally).

The 30-minute default delay window (`SUPERTABLE_GC_DELAY_SEC`) is
chosen so that:

- Any reasonable interactive query (≪ 5 min) is done.
- Long analytical / Spark queries (occasionally 10–30 min) are done.
- The 30-min window is far shorter than the orphan-file accumulation
  rate — Redis stream depth grows at write rate × 30 min, easily
  bounded even at high write velocity.

If you have queries longer than 30 minutes, raise
`SUPERTABLE_GC_DELAY_SEC` to your p99 query duration plus a margin.
Disk usage grows linearly with the delay; choose accordingly.
